========================================================================
  실무와 유사한 구조의 클론 프로젝트 아이디어
========================================================================

  현재 프로젝트 구조:
    프론트(React) → 주문서버(Spring Boot) → 외부 쇼핑몰(NestJS)

  이걸 실무 도메인으로 바꾸면 아래와 같은 프로젝트들이 된다.


========================================================================
[아이디어 1] 이커머스 주문/결제 시스템 (가장 직관적)
========================================================================

  구조:
    프론트(React) → 주문서버(Spring Boot) → 결제서버(NestJS)

  도메인 매핑:
    현재 프로젝트                  클론 프로젝트
    ─────────────────────        ─────────────────────
    챌린지                        상품 카탈로그
    참가 신청                     장바구니 담기
    제출                          주문 요청
    승인 → shop-api 주문          주문 확정 → PG 결제 요청
    order_id 저장                 결제 transaction_id 저장
    APPROVED                      PAID

  재현되는 실무 문제:
    - 결제 더블클릭 → 중복 결제 (SELECT FOR UPDATE)
    - 재고 선착순 → 초과 판매 (원자적 UPDATE)
    - PG 응답 유실 → 결제됐는지 모름 (멱등키)
    - 결제 후 주문 DB 장애 → 유령 결제 (Saga)


================================================================================
[아이디어 2] 선착순 쿠폰/이벤트 시스템 (동시성 집중)
================================================================================

  구조:
    프론트(React) → 이벤트서버(Spring Boot) → 쿠폰서버(NestJS)

  흐름:
    이벤트 참여 신청 → 쿠폰 발급 요청 → 쿠폰서버에서 발급 + 잔여수량 차감

  재현되는 실무 문제:
    - 1000명 동시 접속, 쿠폰 100장 → 초과 발급 방지
    - 같은 사용자 중복 발급 방지
    - 쿠폰 발급 성공했는데 이벤트서버 기록 실패
    - 트래픽 폭주 대응 (Redis 큐 도입 가능)

  추가 기술:
    - Redis로 선착순 카운터 (DECR + 0 이하 차단)
    - 대기열 시스템 (Bull Queue 등)


================================================================================
[아이디어 3] 예약 시스템 (병원/식당/공연)
================================================================================

  구조:
    프론트(React) → 예약서버(Spring Boot) → 알림/결제서버(NestJS)

  흐름:
    시간대 선택 → 예약 요청 → 결제 → 알림 발송

  재현되는 실무 문제:
    - 같은 시간대 동시 예약 → 중복 예약 (락)
    - 좌석/슬롯 한정 → 초과 예약 (원자적 차감)
    - 결제 실패 시 예약 롤백 (Saga)
    - 예약 확정 후 알림 실패 → 재시도 (멱등성)

  추가 기술:
    - 좌석 배치도 UI (프론트 난이도 ↑)
    - 시간대별 슬롯 관리


========================================================================
[아이디어 4] 포인트/머니 충전/송금 시스템 (핀테크)
========================================================================

  구조:
    프론트(React) → 계좌서버(Spring Boot) → 외부 은행 API(NestJS)

  흐름:
    송금 요청 → 출금 계좌 차감 → 외부 은행 API로 입금 → 결과 기록

  재현되는 실무 문제:
    - 동시 송금 → 잔액 음수 (원자적 차감)
    - 출금 성공 + 입금 실패 → 돈 증발 (Saga 필수)
    - 네트워크 유실 → 이중 송금 (멱등키)
    - 동시 충전 + 동시 출금 → 잔액 정합성 (락)

  추가 기술:
    - 거래 원장 (이중 기록 원칙)
    - 정산 배치

================================================================================
[공통 구조]
================================================================================

  어떤 아이디어든 핵심 구조는 동일하다:

    프론트(React) → 메인서버(Spring Boot) → 외부서버(NestJS)
                         │                        │
                    메인 DB (PostgreSQL)     외부 DB (PostgreSQL)

  그리고 해결해야 할 문제도 동일하다:
    1. 중복 요청 방지 (락)
    2. 한정 리소스 경합 (원자적 처리)
    3. 외부 호출 실패 처리 (상태 분리, 재시도)
    4. 응답 유실 대응 (멱등키)
    5. 부분 성공 보상 (Saga/Outbox)

  도메인만 바뀔 뿐, 기술적 챌린지는 지금 하고 있는 것과 같다.
